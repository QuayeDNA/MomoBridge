How it works

Four steps. Zero new infrastructure.

Nothing about how your customer pays changes. They still send Mobile Money the way they always have. The difference is what happens the moment that SMS lands.

STEP 01

SMS arrives

Your phone gets the usual "Confirmed. You have received GHS18.00…" text from MTN, Telecel, AT or T-CASH.

STEP 02

App reads it

MoMo Bridge pulls out the reference number and amount, and saves it on your phone as PENDING.

STEP 03

Customer checks out

On your website, they type in the reference number from their SMS. Your site asks MoMo Bridge: "did this come in?"

STEP 04

Instant answer

Your phone checks its own records and replies CONFIRMED or tells you exactly why not.

For developers

Three ways to drop it into checkout

One script, no dependencies, no build step. Pick the mode that matches your checkout flow — all three talk to the same relay and the same key.

Best for a "Verify Payment" button — opens a centered modal, customer pastes their reference, done.

<script src="https://momo-bridge.vercel.app/momobridge.js"></script> <script> MoMoBridge.popup({ relayUrl: 'https://your-relay.onrender.com', apiKey: 'mb_...', amount: 18.00, title: 'Verify your payment', onSuccess: function(data) { creditWallet(data); }, onFailure: function(data) { showError(data.message); } }); </script>

Renders the verification form directly into a container on your page — no modal, stays put.

MoMoBridge.inline({ relayUrl: 'https://your-relay.onrender.com', apiKey: 'mb_...', amount: 18.00, container: '#verify-container', onSuccess: function(data) { creditWallet(data); }, onFailure: function(data) { showError(data.message); } });

Opens a dedicated verification page. Good for mobile checkouts or when you'd rather not embed any UI at all.

MoMoBridge.redirect({ relayUrl: 'https://your-relay.onrender.com', apiKey: 'mb_...', amount: 18.00, callbackUrl: 'https://myshop.com/checkout', target: '_blank' }); // Your checkout page listens for the result: window.addEventListener('message', e => { if (e.data.type === 'momo_result' && e.data.confirmed) { creditWallet(e.data); } });

No widget at all — call the relay directly from your own backend or frontend.

curl -X POST https://your-relay.onrender.com/claim \ -H "Content-Type: application/json" \ -d '{"apiKey":"mb_...","reference":"0000013331054115","amount":18}'
FieldTypeDescription
apiKeyrequiredstringThe store's key, generated on the phone. Routes the request to the right device.
referencerequiredstringThe transaction reference the customer read off their SMS.
amountrequirednumberExpected amount. Must match the SMS within GH₵0.01.
callbackUrloptionalstringWhere the widget redirects after a result (redirect mode only).

The response always tells you exactly what happened:

CONFIRMED

Reference found, amount matched, marked confirmed.

ALREADY_CONFIRMED

Someone already claimed this reference.

AMOUNT_MISMATCH

Reference exists but the amount sent doesn't match.

EXPIRED

The reference passed its expiry window unclaimed.

INVALID

No transaction with that reference exists yet.

PHONE_OFFLINE

The vendor's phone isn't connected to the relay right now.

Getting started

From install to first confirmed payment

Two pieces to stand up: the app on your phone, and a relay server it talks to. Both take a few minutes.

1 Set up the app

  1. Install the APK and open MoMo Bridge.
  2. Generate an API key — one per store or location.
  3. Point it at your relay URL (default works out of the box).
  4. Select which SMS senders to monitor — MTN, T-CASH, Telecel, AT.
  5. Done. The app starts watching for incoming payments immediately.

2 Deploy your own relay

  1. The relay is one file, zero npm dependencies.
  2. Use the shared default, or deploy your own for full control.
# Deploy to Render, Fly.io, or any box that runs Bun git clone https://github.com/QuayeDNA/MomoBridge cd MomoBridge/relay bun install bun run index.ts # health check curl https://your-relay.com/health

3 Configure SMS parsing

  1. Pick a sample SMS from your inbox as a template.
  2. Confirm the reference, amount, and sender name it detected.
  3. Validate against a second message before saving.
  4. If a carrier changes its format later, the app notices after three failed parses and asks you to retrain it — no code, no redeploy.

4 Embed on your site

  1. Add the momobridge.js script tag to checkout.
  2. Call popup(), inline(), or redirect() with your relay URL and key.
  3. Listen for the result and credit the customer's wallet.

Questions

Things people ask before they install it

Do I need a backend or database?▾
No. Your phone is the database. The relay server only passes messages between your phone and your website — it never stores a transaction.
What happens if my phone loses internet?▾
SMS reading and storage keep working with no connection at all. Only the final confirmation step needs the relay — and if it's offline, customers see an honest "phone offline" message instead of a wrong answer.
Can I run more than one store from one phone?▾
Yes. Generate one API key per store from the same app. Every confirmed transaction records which key claimed it, so you get per-store reporting for free.
What if MTN or Telecel change their SMS wording?▾
The app tracks failed parses per sender. After three in a row, it notifies you with the exact message that failed, pre-filled into the retraining screen — one tap to fix it, no app update required.
Is my customer's data sent anywhere?▾
Only the claim request itself — a reference number and an amount — passes through the relay, and it's never written to disk there. Everything else stays on your phone.
What if my device is lost or stolen?▾
Treat the API key like a password — whoever has it can submit claims. Revoke the key immediately from another device's relay dashboard, or reinstall and generate a fresh one.
